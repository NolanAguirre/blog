The Right Tech Debt

Development February 10, 2026

The premise is this: the right tech debt is debt with a small blast radius. Tech debt in implementation is far less painful than tech debt in interfaces, because implementation debt can usually be paid down locally. Interface debt creates coordination costs between multiple parts of the system, potentially cascading into failures, rewrites, and complexity across the codebase.

Technical debt is not just measured by how ugly the code looks. It is measured by how expensive the code is to change. The safest debt is local. The most dangerous debt is debt that forces unrelated parts of the system to move together.

Implementation Debt Is Usually Contained

When we talk about technical debt, not all debt is created equal. There is a crucial distinction between debt hidden inside a module and debt that affects how components talk to each other.

Implementation debt is contained. You can refactor the internals of a function, rewrite an algorithm, or restructure private data without touching anything else. The blast radius is small.

This kind of debt can still be annoying. It can still slow you down. It can still make a section of code unpleasant to work in. But it is usually repayable by one person, in one place, without requiring the rest of the system to coordinate around the change.

Interface Debt Is Coordination Debt

Interface debt is viral. Change a function signature, modify an API response, rename a field, change an event shape, or alter a database contract, and suddenly you are coordinating across call sites, services, tests, deployments, and sometimes teams.

This is where tech debt becomes expensive. The problem is not only that the code needs to change. The problem is that multiple parts of the system need to change together, and every extra dependency increases the chance that something gets missed.

Interface debt does not just make code harder to change. It makes people harder to coordinate.

Tactical Tech Debt Secures the Future

Tactical tech debt is a form of implementation tech debt that is acquired in order to ship fast without compromising the future. It does not stand in the way of iteration or refactoring. It is the best kind of tech debt to acquire.

The main example of this form of tech debt is debt that secures the future, or safely allows for velocity without forcing the cost of cleanup before the cleanup has value. The clearest example I can think of is implementing a new library function after a pattern has revealed itself, but not updating all code to use this new library immediately.

In doing this, you can use the library for new feature development, gain velocity, and secure the future without incurring the cost of refactoring the broader codebase. You have created the better path forward, but you have not forced the entire system to migrate all at once.

There are three core principles at work with this approach: "If it ain't broke, don't fix it", "secure the future", and "incremental refactor".

If you revisit areas of the codebase for new feature development or support, and they need to be updated to the new library function, then you update that code to leverage the library. The code was changing anyway, so the potential blast radius and testing for the change remain largely the same. You still need to validate the other code changes you made to that portion of the codebase, so you validate the update to the library function in passing.

This is the important distinction: you are not refusing to pay down debt. You are refusing to pay it down when the payment creates more risk than value. The better pattern exists, new work can use it, and old work can move over when it is already being touched.

Some Boundaries Are Worth Abstracting Early

the approach above is not universally applicable. Some changes do require a full migration. Updating database connections to use SSL, for example, might need every connection to move through the new path.

The kinds of changes that require this sort of full refactor should be abstracted from the beginning, even if the abstraction is just wrapping the default behavior in a pass-through function. By doing this, it enables you to make the change in one place and fully leverage the upside of abstracted code.

I would argue that most important I/O operations should be abstracted in this way. HTTP, DB, FS, queues, vendors, and several other interactions should be built on abstractions you control from the outset.

This does not need to be heavy. Taking the time to wrap these libraries in a function that you control is very lightweight, but it can massively pay off down the road. These are the boundaries where future policy changes tend to happen: SSL, retries, timeouts, tracing, auth, rate limits, error handling, testing, and operational behavior.

The goal is not abstraction for its own sake. The goal is to control the parts of the system where change will be expensive later.

The Right Debt Has a Small Blast Radius

The right tech debt lets you ship now without forcing future coordination across unrelated systems. It is local, repayable, and does not corrupt the interfaces other code depends on.

The wrong tech debt leaks into boundaries. It turns simple changes into coordinated migrations. It makes the system harder to move because too many parts need to move at the same time.

Implementation debt can often be lived with. Interface debt compounds. The difference is not cleanliness. The difference is blast radius.
